# Public Docs Boundary and Release Consistency Design

**Date:** 2026-03-07

## Goal

Keep competitor-analysis content out of the public LoopForge docs site while preserving it as internal repository knowledge, and tighten release rules so workspace version, changelog, and Git tag stay aligned.

## Scope

- Remove competitor-analysis pages from the public MkDocs site.
- Preserve those pages under an internal-only repository path.
- Remove public navigation/index references to competitor names.
- Add an automated guard that fails release checks if public docs mention competitor-analysis terms.
- Extend release checks to validate semver tag consistency against the workspace version and HEAD tags when present.
- Document the rules in internal repo docs.

## Non-Goals

- No changes to `openfang` or `.tmp/openclaw`.
- No redesign of the public docs theme.
- No forced Git tag creation in this iteration.

## Recommended Approach

### 1. Public/Internal content split

Move competitor-analysis blog posts out of `docs-site/` into `docs/internal/competitive/`.
That keeps the material available for internal product and positioning review, but it will no longer be generated by MkDocs or linked from the public docs site.

### 2. Public docs guard in existing release check

Reuse `loopforge release check` rather than creating a second policy command.
The release preflight already checks version/changelog/workflow readiness, so it is the right place to fail when public docs accidentally include internal competitor-analysis content.

### 3. Git tag consistency as a release-check rule

Keep the existing `--tag vX.Y.Z` check, and add one more rule:
if the current HEAD already has semver tags, the requested release tag must match them exactly.
That catches mismatched or stale tags without blocking pre-tag dry runs.
